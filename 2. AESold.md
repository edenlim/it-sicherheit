# NIST Kryptostandard
- NIST (National Institute of Standards and Technology) is a US authority that defines official cryptographic standards.
- A NIST cryptographic standard is an algorithm approved for secure use in government and industry.
- Example: AES (Advanced Encryption Standard) is a NIST standard for symmetric encryption.

# AES (Advanced Encryption Standard)

- A symmetric block cipher standardized by NIST.
- Works on 128-bit blocks with key sizes of 128, 192, or 256 bits.
- Uses a structure based on substitution and permutation (SP-network), not Feistel.
- Designed to be fast, secure, and efficient in hardware/software.

Made up of repeating lsayers

- Key addition
- Byte substitution
- Diffusion
    - Shiftrows
    - Mix Columnes

# Feistel Cipher vs AES

- What is a Feistel cipher?
- A structure used in many block ciphers (e.g. DES)
- Splits data into two halves: L (left) and R (right)
- Key idea: encryption and decryption use the same structure

Each round:
$$
L_{i+1} = R_i\\
R_{i+1} = L_i ⊕ F(R_i, K_i)
$$

Why AES is NOT a Feistel cipher
- AES does not split into left/right halves
- Instead it operates on a full state matrix (4×4 bytes)
- Uses transformations:
- SubBytes (substitution)
- ShiftRows (permutation)
- MixColumns (mixing)
- AddRoundKey (XOR with key)

# Chapter 1: The Math Behind AES

Before exploring how AES operates on large files, the lecture explains the complex math required for AES to function securely.

## Finite Fields (Galois Fields)
AES uses a specific mathematical "space" called a Finite Field, specifically `GF(2^8)` or `GF(256)`.

- Unlike normal math where numbers can grow infinitely large, math in this field always results in a number between **0 and 255**.
- This is crucial because it allows every mathematical operation to fit perfectly into a single **8-bit byte**.

## Polynomials instead of Numbers
To perform calculations like multiplication and division within this restricted 8-bit space without errors:

- Bytes are not treated as normal numbers.
- Instead, they are represented as **polynomials**.
- A byte's binary bits (1s and 0s) determine which parts of a polynomial equation are active.

## The S-Box (Substitution Box)
One of the most important parts of AES is substituting one byte for another to create **confusion**.

- The values in the S-Box are generated using complex division (inversion) within the Finite Field.
- Because doing this math live is slow, the results are **pre-calculated** and stored in a lookup table.

# Chapter 2: AES Operating Modes (Betriebsmodi)

AES only encrypts data in fixed **128-bit blocks**. Operating modes define how to securely chain multiple blocks together.

## Electronic Code Book (ECB) — The Unsafe Mode

**How it works:**
- Every 128-bit block is encrypted individually using the same key.

**The Problem:**
- Identical plaintext blocks produce identical ciphertext blocks.
- This leaks patterns (e.g., image structures remain visible).
- Attackers can exploit this to manipulate data.

## Cipher Block Chaining (CBC) — A Safer Mode

**How it works:**
- Each block is XORed with the previous encrypted block before encryption.
- The first block uses a random **Initialization Vector (IV)**.

**The Benefit:**
- Identical plaintext blocks produce completely different ciphertext.
- Patterns are hidden.

## Output Feedback (OFB) — The Stream Mode

**How it works:**
- AES generates a stream of pseudo-random data.
- This stream is XORed with plaintext to produce ciphertext.

# Chapter 3: Hash Functions

A cryptographic hash function acts like a digital **fingerprint**.

## How it works
- Accepts input of any size (text, files, videos, etc.).
- Always outputs a fixed-size, seemingly random string (**hash**).

## The "One-Way" Rule
- Easy to compute a hash from data.
- Practically impossible to reverse the hash back to the original data.

## Collision Resistance
- A secure hash function ensures:
  - Two different inputs do **not** produce the same hash.

**Important:**
- Older algorithms like **MD5** and **SHA-1** are broken.
- Modern standard: **SHA-256**

## Salting
- A random value (**salt**) is added to passwords before hashing.

**Why:**
- Prevents identical passwords from having identical hashes.
- Protects against precomputed dictionary attacks (rainbow tables).

# Chapter 4: Message Authentication Code (MAC)

A MAC ensures:
- **Authenticity** (who sent the message)
- **Integrity** (message was not altered)

## How it works
1. Alice and Bob share a secret key.
2. Alice:
   - Combines message + secret key
   - Hashes them → produces MAC
   - Sends message + MAC
3. Bob:
   - Recomputes MAC using his key
   - Compares with received MAC

**If they match:**
- Message is authentic and unchanged.

## HMAC (Hash-based MAC)

Simple concatenation of key + message is insecure.

**HMAC solves this by:**
- Using a structured method with:
  - Inner hash
  - Outer hash

This ensures:
- Secure mixing of key and message
- Resistance to known cryptographic attacks
