# Operating System Security Notes

### Glossary of Jargons
* **Kernel**: The core part of an operating system with complete control over everything in the system.
* **Protection Rings**: Hierarchical security levels supported by CPU hardware to protect data and functionality from faults or malicious behavior.
* **System Call**: A programmatic way in which a computer program requests a service from the kernel of the operating system.
* **Race Condition**: A situation where the behavior of software depends on the relative timing of events, such as the order in which processes reach a specific resource.
* **Shared Library**: A file that is intended to be shared by executable files and further shared library files.

---

## I. Fundamentals of Operating System (OS) Security
The primary objective of OS security is to maintain the isolation of user data and processes, ensuring that users cannot access the resources of others without authorization.

### A. OS Functionality and Architecture
* **Core Responsibilities**: Modern multi-user systems manage processes, memory, file systems, I/O, and users.
* **Layered Structure**: Systems are split into layers, including hardware (CPU, RAM), kernel space (drivers, resource management), and user space (applications like browsers and games).
* **Hypervisors**: An abstraction layer that acts as an "operating system for operating systems," allowing multiple OS instances to run on the same hardware.



### B. Privilege Separation (Protection Rings)
Processors typically support four privilege levels (0–3), where Ring 0 is the most privileged and Ring 3 is the least privileged.
* **Ring 0 (Kernel Mode)**: Offers direct hardware access and unrestricted control over all processes and memory.
* **Ring 3 (User Mode)**: Restricts programs so they have no direct hardware access and can only manage their own processes and memory.
* **Implementation**: Modern systems like Windows, Linux, and macOS primarily utilize only Ring 0 and Ring 3.



---

## II. System Calls: The Gateway to the Kernel
Programs in User Mode require **System Calls** to transition to Kernel Mode to perform privileged operations.

* **Capabilities**: System calls allow for opening, reading, or writing files, and sending or receiving network packets.
* **Implementation**: They can be implemented via processor interrupts (e.g., `int 0x80`) or fast features like `SYSENTER/SYSCALL`.
* **Abstraction**: Users typically call libraries that "wrap" these calls (e.g., `glibc` in Linux or Native API in Windows).

---

## III. Linux Security Model
Linux utilizes a user-centric model focused on User IDs (UIDs) and Groups rather than abstract roles.

### A. Identities and Permissions (Dateirechte)
* **Root (Superuser)**: A special user capable of accessing and changing anything on the system.
* **Storage**: User and group IDs are stored in `/etc/passwd`, while passwords are kept in `/etc/shadow`.
* **Access Matrix**: Permissions are defined for the **Owner**, **Group**, and **Others**.
* **Types**: Rights include **Read (r)**, **Write (w)**, and **Execute (x)**.

### B. Process Privileges
* **Process IDs**: Every process is uniquely identified by a PID.
* **User IDs**: Rights are governed by the UID (owner), EUID (effective ID used for access checks), and SUID (saved ID for privilege restoration).
* **SUID Bits**: These allow a process to start with a different effective ID than the real ID of the user, making them a high-value target for attackers.

## IV. Common Attacks and Vulnerabilities
Operating systems face constant threats; in 2025, there were 4,417 vulnerabilities with a CVSS score > 5 in the Linux kernel.

### A. TOCTOU (Time-of-Check, Time-of-Use)
A Race Condition where an attacker alters a resource between the security check ($t_1$) and the actual use ($t_2$).
* **Mechanism**: Often involves swapping a legitimate file with a symbolic link to a sensitive file like `/etc/passwd`.
* **Countermeasures**: Use **Immutable Bindings**, work with **File Handles** (as Windows does), or rely on the OS to check rights during the `open()` call rather than checking manually with `access()`.

### B. Shell Attacks
These involve injecting control characters (like `;` or `&`) into command strings to manipulate shell behavior.
* **Risk**: Functions like `system()` or `popen()` are particularly vulnerable if user input is not sanitized.
* **Mitigation**: Using restricted shells (e.g., `bash -r`) can limit directory changes and access to environment variables.

### C. Shared Library Attacks (Preloading)
Shared libraries allow code to be used by multiple programs to save disk space and RAM.
* **LD_PRELOAD**: This variable allows a user to "preload" or replace a library with another, potentially hijacking OS functions.
* **Impact**: An attacker could override keyboard input functions to create a **Key-Logger** or redirect victims by overriding DNS resolution.
* **Security**: Today, preloading is no longer permitted for SUID programs.

---
*Summary of Library Types: Static libraries are bound at link-time for faster code, while Dynamic libraries (.so/.dll) are bound at load-time using tables (PLT/GOT) to resolve addresses.*
